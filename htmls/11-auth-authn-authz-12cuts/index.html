웹 애플리케이션 개발 · 보안/백엔드 · 12컷 이미지 스토리 #11

인증과 인가,
“너는 누구냐”와 “무엇을 할 수 있냐”

공항의 ‘여권 검사대’와 ‘탑승 게이트’에 비유해, 신원을 확인하는 절차와 권한을 결정하는 절차를 12컷으로 풀어봅니다.

AuthenticationvsAuthorization

로그인 화면 하나에는 사실 서로 다른 두 가지 질문이 숨어 있습니다. 첫 번째는 “당신은 누구입니까”이고, 두 번째는 “당신은 무엇을 할 수 있습니까”입니다. 이 둘은 같은 것처럼 보이지만 완전히 다른 절차입니다. 이 글에서는 공항이라는 하나의 은유를 끝까지 유지하며, 여권 검사대는 신원 확인(Authentication)을, 탑승 게이트는 접근 권한(Authorization)을 나타내는 12개의 장면을 하나씩 따라가 봅니다.

PART 1 / 3 — 두 질문의 발견 (컷 01–04)
컷 01

너는 누구이고, 어디까지 들어갈 수 있는가

로그인 시스템은 사실 두 개의 질문으로 이루어진다. 이 둘은 같은 것처럼 보이지만 완전히 다른 절차다.

공항 터미널을 위에서 내려다보면 왼쪽에는 여권 검사대가, 그 너머 긴 복도 끝에는 여러 개의 탑승 게이트가 늘어서 있습니다. 검사대를 통과한 사람의 동선이 점선 화살표로 복도를 따라 이어지고, 어떤 게이트는 열려 있지만 어떤 게이트는 차단봉이 내려와 있습니다. 신원을 확인하는 절차와 그 신원으로 어디까지 들어갈 수 있는지를 정하는 절차 — 이 두 가지가 이번 이야기의 전부입니다.

컷 02

정의부터 명확히: 확인하는 것과 허락하는 것

Authentication은 신원을 확인하는 절차이고, Authorization은 확인된 신원에게 무엇을 허락할지 결정하는 절차다.

인증(Authentication)은 “당신이 주장하는 그 사람이 맞는지”를 확인하는 절차입니다. 로그인 시 아이디와 비밀번호, 혹은 생체 정보로 신원을 증명하는 과정이 여기에 해당합니다. 인가(Authorization)는 이미 신원이 확인된 사용자에게 “어떤 자원과 기능에 접근할 권한이 있는지”를 결정하는 절차입니다. 인증 없이는 인가를 판단할 대상 자체가 없으므로, 두 절차는 항상 인증이 먼저이고 인가가 그 뒤를 따릅니다. AuthN(확인) → AuthZ(허락)

컷 03

나란히 비교: 검사대 한 번 vs 게이트마다 확인

인증은 보통 한 번만 통과하면 되는 관문이지만, 인가는 이동할 때마다 매번 다시 확인되는 반복적 절차다.

여권 검사대→ 한 번 통과 → 신원 유지→ 게이트 A✓ 게이트 B✕ 게이트 C✓

인증은 공항 입구의 여권 검사대처럼 여정 초반에 한 번 통과하면 그 신원이 세션이나 토큰 형태로 유지됩니다. 반면 인가는 탑승 게이트마다 티켓 등급을 다시 확인하듯, 사용자가 어떤 자원에 접근하려 할 때마다 매번 개별적으로 검사됩니다. role: admin, role: editor, role: guest 같은 등급 태그가 게이트마다 다시 대조되는 셈입니다. 즉 로그인은 한 번이지만, 권한 검사는 요청마다 반복된다는 점이 두 개념의 실질적인 동작 차이입니다.

PART 2 / 3 — 방식과 동작 원리 (컷 05–08)
컷 05

인증 방식 카탈로그: 신원을 확인하는 네 가지 방법

여권을 확인하는 방법이 여러 가지이듯, 신원을 증명하는 기술적 방법도 세션, 토큰, OAuth, 생체/MFA 등으로 다양하다.

세션 기반 인증은 서버가 로그인 상태를 기억하고 브라우저에 세션 쿠키를 내려주는 방식입니다. 토큰 기반 인증(JWT)은 서버가 서명한 토큰을 클라이언트가 보관하다가 매 요청마다 제시하는 방식입니다. OAuth와 SSO는 구글, 깃허브 같은 제3자가 신원을 대신 보증해주는 방식이고, MFA는 비밀번호 외에 지문이나 OTP 같은 추가 증거를 요구해 인증을 이중으로 강화하는 방식입니다. 네 가지 방법 모두 결국 같은 여권 검사대의 서로 다른 도구일 뿐입니다.

컷 06

동작 원리: 로그인부터 재확인까지의 순환

로그인 요청, 신원 확인, 토큰 발급, 이후 요청마다의 재증명이라는 순환 고리가 인증 시스템의 실제 동작이다.

① 로그인 요청→ ② 서버가 신원 확인→ ③ 토큰/세션 발급→ ④ 요청마다 토큰 제시↺

사용자가 아이디와 비밀번호를 보내면 서버는 저장된 값과 대조해 신원을 확인하고, 성공하면 세션 ID나 서명된 토큰을 발급합니다. 이후 사용자는 매 요청마다 그 세션 쿠키나 토큰을 함께 보내고, 서버는 그것만 보고도 별도의 재로그인 없이 신원을 다시 증명받습니다. login → verify → issue token → re-present 이 순환 구조 덕분에 사용자는 페이지를 이동할 때마다 비밀번호를 다시 입력하지 않아도 됩니다.

컷 07

헷갈리는 경계: 로그인은 됐는데 왜 못 들어가지?

인증에 성공했다고 해서 모든 곳에 접근할 수 있는 것은 아니다. 인증 성공과 인가 실패는 동시에 일어날 수 있다.

⚠ AuthN: OK / AuthZ: DENIED

일반 회원으로 로그인한 사용자가 관리자 전용 페이지에 접근하려 하면, 인증은 이미 통과했음에도 인가 단계에서 거절당합니다. 게이트 옆 표지판에는 “이 구역은 role: admin 등급만 출입 가능”이라 적혀 있지만, 사용자의 배지에는 role: guest가 붙어 있는 셈입니다.

✓ 원인은 대부분 인가 쪽에 있다

“로그인은 되는데 왜 이 기능은 안 되지”라는 질문은 대부분 인증이 아니라 인가의 문제입니다. 로그인에 성공했다는 것은 신원이 확인되었다는 뜻일 뿐, 모든 기능을 사용할 수 있다는 뜻은 아니라는 점을 기억해야 합니다.

실무에서 흔히 발생하는 혼란이 바로 이 지점입니다. 여권 검사대는 이미 통과했지만, 그다음 게이트 앞에서 차단봉을 만나는 경우가 생각보다 자주 일어납니다.

컷 08

코드로 보는 예시: 토큰 발급과 권한 체크

인증은 토큰을 발급하는 코드로, 인가는 그 토큰 안의 역할(role)을 검사하는 미들웨어 코드로 구현된다.

authController.js — 신원 확인 후 토큰 발급
// 로그인 성공 시 서명된 토큰을 발급한다
const token = jwt.sign(
  { userId: user.id, role: user.role },
  SECRET,
  { expiresIn: '1h' }
);
res.json({ token }); // 증명서 발급
roleMiddleware.js — 권한(role) 검사
// 토큰 속 role이 admin인지 검사한다
function requireAdmin(req, res, next) {
  if (req.user.role !== 'admin') {
    return res.status(403).json({
      error: 'Forbidden'
    });
  }
  next(); // 자격 확인, 통과
}

인증은 보통 로그인 라우트에서 사용자 정보를 검증한 뒤 jwt.sign()으로 서명된 토큰을 발급하는 코드로 구현됩니다. 인가는 이후 요청을 가로채는 미들웨어에서 그 토큰을 해독해 얻은 role 값이 필요한 등급과 일치하는지 검사하고, 일치하지 않으면 403 Forbidden을 돌려주는 코드로 구현됩니다. 발급은 인증, 검사는 인가 — 하나는 증명서를 발급하는 코드이고, 다른 하나는 그 증명서로 자격을 검사하는 코드입니다.

PART 3 / 3 — 실수, 선택, 실전 판별 (컷 09–12)
컷 09

흔한 실수: 평문 비밀번호와 빠뜨린 권한 검사

비밀번호를 그대로 저장하는 실수는 인증을 무너뜨리고, 인가 검사를 빠뜨리는 실수는 IDOR 같은 취약점을 낳는다.

⚠ 실수 1 — 비밀번호 평문 저장

비밀번호를 데이터베이스에 그대로 저장하면 서버가 뚫렸을 때 모든 사용자의 비밀번호가 그대로 유출됩니다. 반드시 bcrypt 같은 해시 함수로 되돌릴 수 없게 변환해 저장해야 합니다.

⚠ 실수 2 — 인가 검사 누락

로그인 여부만 확인하고 소유권이나 역할 검사를 빠뜨리면, 사용자가 URL의 id 값만 바꿔 남의 주문 내역이나 게시글에 접근하는 IDOR(Insecure Direct Object Reference) 취약점으로 이어집니다.

✓ 저장은 해싱, 접근은 항상 재검사

인증 확인과 인가 확인은 서로 다른 단계이므로 둘 다 빠짐없이 구현해야 합니다.

컷 10

트레이드오프 비교표: 세션 vs 토큰

세션 기반 인증과 토큰 기반 인증은 서버 부담, 확장성, 로그아웃 처리, 보안 측면에서 서로 다른 장단점을 가진다.

선택은 시스템 구조에 달려 있다
비교 축 세션 기반 (쿠키) 토큰 기반 (JWT)
서버 부담서버가 세션 저장소를 유지서버는 상태를 저장하지 않음
확장성서버 간 세션 저장소 공유 필요각 서버가 독립적으로 검증 가능
로그아웃 처리즉각 무효화 용이만료 전까지 즉각 무효화 어려움
보안 특성신원 정보는 서버 금고에 보관토큰 자체에 정보 포함, 서명 검증 필수

세션 기반 인증은 서버가 로그인 상태를 직접 저장하고 관리하므로 로그아웃이나 강제 만료가 즉각적이지만, 서버가 여러 대로 늘어나면 세션 저장소를 공유해야 하는 확장성 부담이 생깁니다. 토큰 기반 인증(JWT)은 필요한 정보를 토큰 자체에 서명해 담기 때문에 서버가 상태를 저장할 필요가 없어 여러 서버로 쉽게 확장되지만, 한 번 발급된 토큰은 만료 시간이 되기 전까지 즉시 무효화하기 어렵다는 단점이 있습니다. 어느 쪽이 우월한 것이 아니라 시스템 구조에 따른 트레이드오프입니다.

컷 11

언제 무엇을 쓰나: 전통 웹앱 vs 분산 시스템

서버 한 대로 돌아가는 전통적인 웹앱에는 세션이 자연스럽고, 모바일 앱이나 마이크로서비스처럼 여러 서버가 얽힌 구조에는 토큰이 자연스럽다.

컷 12

실전 판별법과 핵심 요약

401은 신원 확인 실패, 403은 권한 부족. 이 한 쌍의 상태코드만 기억해도 실전에서 문제의 원인을 즉시 구분할 수 있다.

401 UNAUTHORIZED
인증 실패 —
너는 누구인지 모르겠다
403 FORBIDDEN
인가 실패 —
너는 알지만 허락할 수 없다
여권을 보여줄 수 없으면 401, 여권은 맞지만 이 게이트는 못 지나가면 403

서버가 401 Unauthorized를 반환하면 이는 “누구인지 확인할 수 없다”는 뜻으로 인증 단계의 실패이고, 로그인을 하지 않았거나 토큰이 만료된 경우입니다. 서버가 403 Forbidden을 반환하면 이는 “누구인지는 알지만 그 행동을 허락할 수 없다”는 뜻으로 인가 단계의 실패이며, 이미 로그인은 되어 있지만 해당 자원에 대한 권한이 없는 경우입니다. 인증은 신원을 확인하는 여권 검사대이고, 인가는 그 신원으로 어디까지 들어갈 수 있는지를 정하는 탑승 게이트라는 이 하나의 그림만 기억해도 두 개념을 혼동하지 않을 수 있습니다.

인증 = 신원 확인 인가 = 권한 확인 인증이 항상 먼저 세션은 서버가 기억, 토큰은 스스로 증명 해싱 없는 저장은 금물 401 vs 403을 구분하라
