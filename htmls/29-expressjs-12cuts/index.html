웹 애플리케이션 개발 · 백엔드 프레임워크 · 12컷 이미지 스토리 #29

Express.js,
Node.js로 서버를 만드는 가장 쉬운 길

벽돌부터 손으로 쌓을 것인가, 이미 세워진 골조에 문만 달 것인가 — 그 차이를 12컷으로 풀어봅니다.

Node.js→Express

서버를 만든다는 것은 결국 무엇으로, 어떻게 지을 것인지를 정하는 일입니다. Node.js만으로도 서버는 만들 수 있지만, 그것은 벽돌 한 장 한 장을 손으로 쌓아 집을 짓는 것과 같습니다. 요청을 받는 방법부터 주소를 구분하는 방법까지 전부 직접 코드로 짜야 하니까요. Express는 이미 기둥과 보가 세워진 조립식 뼈대를 제공해서, 개발자는 그 뼈대에 문(라우트)만 달면 서버가 완성되도록 도와줍니다. 이 글은 그 차이를 열두 장면으로 풀어봅니다.

PART 1 / 3 — 재료와 뼈대의 발견 (컷 01–04)
컷 01

서버를 어떻게 지을 것인가

서버를 만든다는 것은 결국 벽돌부터 하나씩 쌓을 것인가, 이미 세워진 골조에 문만 달 것인가를 정하는 일이다.

화면을 절반으로 나누면 왼쪽엔 아직 쌓이지 않은 그레이 벽돌 더미와 흙손이, 오른쪽엔 기둥과 보가 매끈하게 이어진 Node 그린 철골 뼈대가 서 있습니다. 뼈대의 벽면에는 문을 끼울 수 있는 빈 슬롯이 규격대로 뚫려 있죠. Node.js는 이 벽돌 더미이고, Express는 이미 조립된 저 골조입니다. Node.js만으로도 서버를 만들 수 있지만, 요청을 받는 방법부터 주소를 구분하는 방법까지 전부 직접 코드로 짜야 합니다. Express는 개발자가 그 뼈대에 문만 달면 서버가 완성되도록 도와줍니다.

컷 02

정의부터 명확히: 재료와 뼈대

Node.js는 자바스크립트를 서버에서 실행하는 재료이고, Express는 그 위에 미리 조립된 웹 프레임워크(뼈대)다.

Node.js는 원래 브라우저 안에서만 돌던 자바스크립트를 서버 컴퓨터에서도 실행할 수 있게 해주는 실행 환경입니다. 그 자체로도 웹 서버를 만들 수 있지만, 주소별로 처리를 나누고 요청을 가공하는 기능까지 전부 직접 구현해야 합니다. Express는 이 Node.js 위에서 동작하는 프레임워크로, 라우팅과 미들웨어라는 이미 만들어진 뼈대를 제공해 서버 제작을 훨씬 단순하게 만들어 줍니다. Node.js가 재료라면 Express는 그 재료로 미리 짜 놓은 조립식 골조인 셈입니다. runtime vs framework

컷 03

나란히 비교: 손수 쌓기 vs 뼈대에 문 달기

기본 http 모듈로는 주소 구분과 요청 처리를 모두 수동으로 짜야 하지만, Express는 함수 호출 몇 줄로 이를 대신한다.

Node.js의 기본 http 모듈만으로 서버를 만들면, 들어온 요청의 주소가 무엇인지 매번 if문으로 직접 확인하고, 메서드가 GET인지 POST인지도 손수 분기해야 합니다. 벽돌을 한 장씩 쌓아 벽을 세우는 것과 같습니다. Express를 쓰면 이 모든 과정이 app.get('/users', 처리함수) 같은 한 줄로 정리됩니다. 이미 세워진 뼈대에 ‘이 주소로 오면 이 문을 열어라’는 팻말만 붙이면 되는 셈입니다.

PART 2 / 3 — 도구와 동작 원리 (컷 05–08)
컷 05

도구 카탈로그: 뼈대에 딸린 부속품들

Express는 라우팅 함수, 미들웨어, Router, body-parser, cors 같은 부속품을 함께 제공한다.

app.get, app.post는 특정 주소와 메서드로 들어온 요청을 어떤 함수로 처리할지 정해주는 라우팅 기능입니다. 미들웨어는 요청이 최종 처리 함수에 도달하기 전이나 후에 공통으로 거쳐 가는 통로로, 로그를 남기거나 인증을 확인하는 데 쓰입니다. Router는 관련된 라우트들을 하나의 묶음으로 정리해주고, body-parser는 요청 본문에 담긴 JSON 같은 데이터를 꺼내 쓸 수 있게 해주며, cors는 다른 도메인에서 온 요청을 허용할지 정하는 역할을 합니다. 이 부속품들이 모여 Express라는 뼈대를 완성하지만, 이 모든 부속은 결국 Node.js라는 재료 위에서 작동한다는 사실은 변하지 않습니다.

컷 06

동작 원리: 요청이 뼈대를 통과하는 흐름

요청은 여러 미들웨어를 순서대로 통과한 뒤, 주소에 맞는 라우트 함수에서 최종 응답으로 만들어진다.

HTTP request→ 로그 미들웨어→ 인증 미들웨어→ 본문 해석 미들웨어→ 라우트 처리→ res.json(...)

Express로 만든 서버에 요청이 들어오면, 먼저 app.use로 등록해둔 미들웨어들을 코드에 작성된 순서대로 하나씩 통과합니다. 각 미들웨어는 로그를 남기거나, 로그인 여부를 확인하거나, 요청 본문을 해석하는 등 자기 몫의 작업을 하고 next()를 호출해 다음 단계로 넘깁니다. 모든 미들웨어를 통과한 요청은 마지막으로 주소와 일치하는 라우트 함수에 도달해 실제 처리가 이루어지고, res.json 같은 함수로 응답이 돌아갑니다. 이 순서는 복도를 따라 차례로 문을 통과하는 것과 같습니다.

컷 07

헷갈리는 경계: Express는 Node.js를 대신하지 않는다

Express는 Node.js를 대체하는 것이 아니라 그 위에 얹혀 동작할 뿐이며, 기본기는 여전히 필요하다.

초보자들이 흔히 하는 오해 중 하나는 Express를 배우면 Node.js는 몰라도 된다고 생각하는 것입니다. 하지만 Express는 Node.js라는 재료 위에 세워진 뼈대일 뿐, 파일을 읽고 쓰거나 비동기 작업을 처리하는 Node.js의 기본기는 Express를 쓸 때도 그대로 필요합니다. 뼈대의 기둥 하나가 벽돌 더미 속으로 뿌리처럼 박혀 있는 모습을 떠올려 보세요 — 뼈대는 재료 없이 서지 못합니다. 예를 들어 데이터베이스 조회나 파일 업로드 같은 작업은 여전히 Node.js의 비동기 처리 방식을 이해해야 제대로 다룰 수 있습니다. Express는 이 기본기 위에 편의를 더해주는 도구이지, 기본기 자체를 없애주는 마법은 아닙니다. Express runs on Node.js

컷 08

코드로 보는 차이: 문 하나 다는 데 걸리는 줄 수

기본 http 모듈로는 주소 하나를 처리하는 데도 여러 줄의 분기 코드가 필요하지만, Express로는 한 줄로 끝난다.

Node.js — 벽돌 쌓기
// 주소와 메서드를 직접 비교
const http = require('http');

http.createServer((req, res) => {
  if (req.url === '/users' && req.method === 'GET') {
    res.writeHead(200, { 'Content-Type': 'application/json' });
    res.end(JSON.stringify(users));
  } else {
    res.writeHead(404);
    res.end();
  }
});
Express — 문 달기
// 주소 하나에 처리 함수 하나
const express = require('express');
const app = express();

app.get('/users', (req, res) => {
  res.json(users); // 끝
});

Node.js 기본 http 모듈로 /users 주소를 처리하려면, 요청 객체의 url 값을 직접 비교하는 if문을 써야 하고 메서드 구분까지 손수 챙겨야 합니다. Express에서는 app.get('/users', (req, res) => res.json(users)) 한 줄이면 같은 일을 끝낼 수 있습니다. 코드가 짧아졌다고 기능이 줄어든 것이 아니라, 반복되는 분기 처리를 Express가 뼈대 안에 이미 갖추고 있기 때문입니다.

PART 3 / 3 — 실전 감각과 요약 (컷 09–12)
컷 09

흔한 실수: 문 열기 전에 경비를 세워야 하는데 늦게 세운 뼈대

인증 미들웨어를 라우트보다 뒤에 등록하면, 이미 문이 열려버린 뒤에 경비를 세우는 셈이라 인증 없이도 요청이 처리된다.

⚠ app.use(인증)이 app.get(라우트)보다 늦게 등록됨

Express의 미들웨어는 코드에 등록된 순서대로 실행됩니다. 만약 인증을 확인하는 미들웨어를 app.use로 등록하기 전에 이미 해당 라우트를 app.get으로 먼저 등록해버리면, 요청은 인증 확인을 거치지 않고 그대로 라우트 함수에 도달해버립니다. 손님이 이미 문을 열고 들어간 뒤에야 경비원을 세우는 것과 같아서, 사실상 검문이 아무 의미가 없어집니다.

✓ app.use(인증) → app.get(라우트) 순서로 등록

인증이나 로그 같은 공통 미들웨어는 반드시 관련 라우트보다 먼저 등록해야 합니다. Express에서 순서는 곧 실행 순서라는 점을 항상 기억해야 합니다.

경비원이 복도 입구에 먼저 서 있고 그 뒤에 라우트 문이 있어야, 손님이 지나갈 때마다 제대로 된 검문이 이루어집니다. ‘순서가 곧 실행 순서’라는 사실 하나만 기억해도 이런 실수는 대부분 피할 수 있습니다.

컷 10

트레이드오프 비교표: 빠른 조립과 스스로 챙겨야 할 정리

Express는 적은 코드로 서버를 빠르게 만들 수 있게 해주지만, 구조를 스스로 정리하지 않으면 관리가 어려워진다.

same runtime, different amount of scaffolding
비교 축 Node.js 기본 http 모듈 Express
주소 하나 처리하는 코드량많음 — if문으로 직접 분기적음 — app.get 한 줄
정해진 구조 규칙없음 — 전부 자유없음 — 강제 규칙은 없지만 라우팅 관례는 제공
초기 체감 난이도높음 — 기초부터 직접 구현낮음 — 바로 라우트 작성 가능
장기 유지보수 리스크낮음 — 구조가 단순해 볼 게 적음있음 — 정리 안 하면 라우트·미들웨어가 뒤엉킴
대표 상황초경량 서버, 학습용실무 REST API, 빠른 프로토타이핑

Express는 최소한의 규칙만 강제하는 가벼운 프레임워크여서, 몇 줄의 코드만으로도 금방 동작하는 서버를 만들 수 있습니다. 하지만 이 가벼움은 동시에 ‘정해진 폴더 구조나 규칙이 없다’는 뜻이기도 합니다. 라우트를 한 파일에 계속 몰아 적거나 미들웨어를 순서 없이 여기저기 등록하다 보면, 프로젝트가 커질수록 어떤 문이 어디로 연결되는지 한눈에 파악하기 어려워집니다. 자유로움에는 책임이 따릅니다.

컷 11

실전 활용 팁: 뼈대를 구역별로 정리하기

라우트를 기능별 파일로 나누고, 미들웨어 등록 순서를 주기적으로 점검하는 것이 Express 프로젝트를 오래 건강하게 유지하는 방법이다.

🧱 라우트를 구역별로 나누기

  • routes/users.js, routes/posts.js처럼 기능별 파일 분리
  • 관련된 라우트는 Router로 묶어 관리
  • 폴더 구조만 봐도 어떤 문이 어디로 연결되는지 파악

🚪 미들웨어 순서 점검하기

  • app.use 등록 순서를 정기적으로 확인
  • 로그 → 인증 → 본문 해석 → 라우트 순서 유지
  • 인증 미들웨어는 관련 라우트보다 먼저 등록

프로젝트가 커지면 모든 라우트를 한 파일에 몰아넣기보다, users, posts, auth처럼 기능별로 파일을 나누고 Express의 Router로 각각을 묶어 관리하는 것이 좋습니다. 또한 서버가 예상과 다르게 동작할 때는 app.use로 등록한 미들웨어들이 어떤 순서로 쌓여 있는지부터 다시 점검하는 습관을 들이면, 인증이 빠졌거나 로그가 누락되는 실수를 미리 잡아낼 수 있습니다.

컷 12

판별법과 핵심 요약: 뼈대가 서 있는지 확인하는 법

req.url을 직접 비교하는 코드가 보이면 기본 http 모듈, app.get·app.use가 보이면 Express다.

가장 간단한 판별법은 이것입니다. 코드 안에 req.url을 직접 확인하고 if문으로 주소를 분기하는 모습이 보인다면 Node.js 기본 http 모듈만 쓴 것이고, app.get이나 app.use 같은 함수가 보인다면 그것은 Express입니다. Node.js는 서버를 지을 수 있는 재료이고, Express는 그 재료 위에 이미 세워둔 조립식 뼈대여서 라우팅과 미들웨어만 신경 쓰면 문을 달듯 서버를 완성할 수 있습니다. 다만 뼈대가 가볍고 자유로운 만큼, 라우트를 정리하고 미들웨어 순서를 챙기는 일은 여전히 개발자의 몫입니다.

Node.js는 재료, Express는 뼈대 라우팅과 미들웨어가 핵심 순서가 곧 실행 순서 가벼운 만큼 구조는 스스로 정리 라우트는 파일별로 분리 app.get이 보이면 Express
재료와 뼈대, 하나의 서버

Node.js는 서버를 지을 수 있는 재료이고, Express는 그 위에 이미 세워진 조립식 뼈대입니다.
벽돌을 손으로 쌓든 뼈대에 문을 달든, 결국 완성되는 것은 하나의 서버입니다.
