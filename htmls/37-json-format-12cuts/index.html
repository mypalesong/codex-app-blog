PART 1 / 3 — 공통 서식의 발견 (컷 01–04)
컷 01

서로 다른 나라 말을 어떻게 이해할까

프론트엔드와 백엔드는 서로 다른 프로그래밍 언어로 만들어질 수 있지만, 데이터를 주고받을 때는 둘 다 이해할 수 있는 공통 서식이 필요하다.

파란 깃발의 프론트엔드국과 초록 깃발의 백엔드국이 마주 선 사이, 공중에는 서류 한 장이 떠 있습니다. 용지 위에는 중괄호 { }가 도장처럼 큼직하게 찍혀 있고, 양쪽에서 뻗어온 손이 그 서류를 함께 붙잡고 있습니다. 프론트엔드가 자바스크립트로, 백엔드가 파이썬이나 자바로 만들어져도 둘은 결국 데이터를 주고받아야 합니다. 서로 다른 나라에서 온 사람들이 대화하려면 공통 서식이 필요하듯, 서로 다른 언어로 만들어진 시스템끼리도 데이터를 담는 공통 형식이 필요합니다. 이 시리즈는 그 공통 서식, JSON — JavaScript Object Notation이 무엇이고 왜 쓰이는지를 하나씩 풀어봅니다.

컷 02

정의부터 명확히: 만국 공통 서식 용지

JSON은 키와 값의 쌍으로 데이터를 표현하는 텍스트 기반의 데이터 교환 형식이다. 사람도 읽을 수 있고 대부분의 프로그래밍 언어가 이해할 수 있다.

JSON은 JavaScript Object Notation의 줄임말로, 이름(key)과 값(value)을 쌍으로 묶어 데이터를 표현하는 텍스트 형식입니다. 이름은 항상 큰따옴표로 감싼 문자열이고, 값에는 문자열, 숫자, 참/거짓, 배열, 다른 객체까지 들어갈 수 있습니다. "name"이라는 라벨 옆에 값이 채워지는 서식 칸을 떠올려 보세요. 특정 프로그래밍 언어의 전용 포맷이 아니라 순수한 텍스트이기 때문에, 자바스크립트든 파이썬이든 자바든 거의 모든 언어가 이 서식을 읽고 쓸 수 있습니다. key-value text format

컷 03

나란히 비교: 장황한 서류 vs 간결한 서식

XML은 여닫는 태그로 데이터를 감싸 장황하고 복잡해지기 쉬운 반면, JSON은 중괄호와 콜론만으로 같은 정보를 훨씬 간결하게 표현한다.

XML — 장황한 서류
<user>
  <name>Song</name>
  <age>28</age>
</user>
JSON — 간결한 서식
{ "name": "Song", "age": 28 }

XML은 데이터마다 여는 태그와 닫는 태그를 짝지어 감싸야 하기 때문에, 정보가 늘어날수록 문서가 장황해지고 눈으로 읽기도 번거로워집니다. JSON은 중괄호 { }로 하나의 데이터 묶음을 표시하고 콜론 :으로 이름과 값을 잇기만 하면 되므로 같은 정보를 훨씬 짧고 간결하게 표현할 수 있습니다. 오늘날 웹 API에서 XML보다 JSON이 훨씬 널리 쓰이는 이유 중 하나가 바로 이 간결함입니다.

PART 2 / 3 — 구조와 동작 원리 (컷 05–08)
컷 05

관련 개념 카탈로그: 서식 용지의 부품들

JSON을 이루는 기본 부품에는 key-value 쌍, 여러 값을 나열하는 배열, 데이터 안에 데이터를 품는 중첩 객체, 그리고 이를 다루는 변환 함수와 오류가 있다.

JSON의 가장 작은 단위는 이름과 값이 짝을 이루는 key-value 쌍이고, 여러 값을 순서대로 나열할 때는 대괄호 [ ]로 묶은 배열을 씁니다. 데이터 안에 또 다른 데이터 묶음이 필요할 때는 중괄호 { } 객체를 그 안에 다시 품는 중첩 객체를 만들 수 있습니다. 프로그램 안의 데이터와 JSON 텍스트 사이를 오갈 때는 JSON.stringify로 텍스트로 바꾸고 JSON.parse로 다시 데이터로 되돌리며, 이 서식의 규칙을 어기면 SyntaxError라는 오류로 알려줍니다.

컷 06

동작 원리: 서식 용지가 오가는 흐름

백엔드는 프로그램 안의 데이터를 JSON 텍스트로 바꿔 보내고, 프론트엔드는 그 텍스트를 받아 다시 프로그램이 다룰 수 있는 데이터로 되돌린다.

백엔드 객체→ JSON.stringify→ application/json→ JSON.parse→ 프론트엔드 객체

백엔드는 데이터베이스에서 꺼낸 프로그램 내부의 객체를 그대로 네트워크로 보낼 수 없기 때문에, JSON.stringify로 이를 하나의 텍스트, 즉 JSON 문자열로 접어서 내보냅니다. 이 텍스트는 네트워크를 통해 프론트엔드로 전달되며, 흔히 응답 헤더에 Content-Type: application/json이라는 표시가 함께 붙습니다. 프론트엔드는 이 텍스트를 받아 JSON.parse로 다시 자바스크립트 객체로 펼쳐서 화면에 표시할 수 있는 데이터로 되돌립니다.

컷 07

헷갈리는 경계: 서식 용지와 진짜 서류철

JSON 텍스트 자체와, 프로그래밍 언어 안에서 그 텍스트를 해석해 만든 객체는 서로 다른 것이다. 이 둘을 혼동하면 오류의 원인을 찾기 어려워진다.

JSON은 어디까지나 글자로 이루어진 문자열이며, 그 자체로는 .name처럼 값을 바로 꺼낼 수 없습니다. 이 문자열을 JSON.parse로 해석해야 비로소 자바스크립트 객체가 되어 프로그램 안에서 값을 자유롭게 꺼내 쓸 수 있습니다. 반대로 객체를 다시 텍스트로 만들 때는 JSON.stringify를 씁니다. 서식 용지 자체(텍스트)와 그 서식을 보고 정리한 서류철(객체)은 비슷해 보여도 다루는 방법이 다르다는 점을 꼭 구분해야 합니다. 둘을 헷갈리면 .name이 안 먹힙니다.

컷 08

코드로 보는 예시: 서식 용지 작성과 해독

JSON 문자열을 만드는 코드와 그 문자열을 다시 데이터로 읽어내는 코드는 짝을 이루어 함께 쓰인다.

서식 용지 작성하기
const data = { name: "Song", age: 28 };
const json = JSON.stringify(data);
// 데이터를 텍스트로 접는다
서식 용지 해독하기
const obj = JSON.parse(json);
console.log(obj.name); // "Song"
// 텍스트를 다시 데이터로 편다

JSON.stringify(data)는 자바스크립트 객체 data를 JSON 문자열로 접어서, 서버로 보내거나 저장할 수 있는 텍스트로 만들어 줍니다. 이렇게 만들어진 문자열을 받은 쪽에서는 JSON.parse(json)으로 다시 원래의 객체 형태로 펼쳐냅니다. 이렇게 펼쳐진 obj는 평범한 자바스크립트 객체이므로 obj.name처럼 점 표기법으로 값을 바로 꺼내 쓸 수 있습니다.

PART 3 / 3 — 실전 감각과 요약 (컷 09–12)
컷 09

흔한 실수: 잘못 찍힌 도장, 어긋난 따옴표

JSON 서식 용지에 문자열 안 따옴표를 잘못 쓰면(작은따옴표 사용 등) 문법 오류가 발생해 전체 서식이 무효 처리된다.

⚠ { 'name': 'Song' } — SyntaxError: Unexpected token

JSON 규격에서는 이름과 문자열 값을 반드시 큰따옴표로 감싸야 하며, 자바스크립트에서는 자연스러운 작은따옴표(')를 쓰면 문법 오류(SyntaxError)가 발생합니다. 사람이 보기엔 멀쩡해 보여도 엄격한 JSON 서식 규칙을 어긴 것이라 JSON.parse가 실패합니다.

✓ { "name": "Song" } — double quotes only

이런 실수는 특히 코드 안에서 JSON 문자열을 직접 손으로 작성할 때 자주 발생하므로, 문자열 값은 항상 큰따옴표로만 감싸야 한다는 규칙을 기억해야 합니다.

삐뚤게 찍힌 “SyntaxError” 도장 하나가 서식 용지 전체를 무효로 만듭니다. 따옴표 하나의 차이가 전체 데이터 교환을 멈춰 세울 수 있다는 사실을 늘 염두에 두어야 합니다.

컷 10

트레이드오프 비교표: 가벼운 서식의 대가

JSON은 사람도 읽기 쉽고 대부분의 언어가 지원해 범용성이 높지만, 아주 큰 데이터를 다룰 때는 다른 이진 포맷보다 용량이 커질 수 있다.

readable and universal, but not always the lightest
비교 축 JSON의 장점 JSON의 한계
가독성사람이 텍스트만 봐도 구조를 바로 이해—
언어 지원JS·Python·Java 등 사실상 모든 주류 언어 지원—
데이터 용량—큰 데이터일수록 이진 포맷보다 용량 증가
구조 표현력중첩 객체·배열로 복잡한 구조도 표현—

JSON의 가장 큰 장점은 사람이 텍스트만 봐도 구조를 바로 이해할 수 있고, 자바스크립트뿐 아니라 파이썬, 자바 등 사실상 모든 주류 언어에서 표준 라이브러리로 다룰 수 있다는 범용성입니다. 다만 이름과 구두점까지 모두 글자로 표현하는 텍스트 형식이다 보니, 데이터 양이 아주 많아지면 이진(binary) 형식의 포맷보다 전송 용량이 커질 수 있습니다. 그래서 일반적인 API 응답에는 JSON이 충분히 가볍고 실용적이지만, 초대용량 데이터를 다루는 특수한 상황에서는 다른 포맷이 고려되기도 합니다.

컷 12

판별법과 핵심 요약: 도장이 찍힌 서식인가

큰따옴표로 감싼 이름과 값, 중괄호나 대괄호로 정확히 여닫힌 구조를 갖췄다면 유효한 JSON이고, 그렇지 않으면 서식 오류다.

가장 간단한 판별법은 이것입니다. 모든 이름과 문자열 값이 큰따옴표로 감싸여 있고, 중괄호와 대괄호가 정확히 열고 닫히며, 마지막 항목 뒤에 불필요한 쉼표가 없다면 유효한 JSON입니다. JSON은 프론트엔드와 백엔드가 서로 다른 언어로 만들어졌더라도 데이터를 오해 없이 주고받게 해주는 만국 공통 서식 용지이며, JSON.stringify와 JSON.parse라는 두 개의 도구로 그 서식을 접고 펴며 대화를 이어갑니다. 이 규칙 하나만 몸에 익히면 API를 다루는 모든 순간이 훨씬 수월해집니다.

JSON = key-value 텍스트 서식 XML보다 간결하다 stringify로 접고 parse로 편다 텍스트와 객체는 다르다 작은따옴표는 오류의 원인 판별법: 큰따옴표 + 짝 맞는 괄호
하나의 서식 용지, 두 개의 나라

프론트엔드국과 백엔드국은 서로 다른 언어를 쓰지만, 그 사이에서 오가는 서식 용지 한 장이 있어 오해 없이 데이터를 주고받습니다.
큰따옴표로 감싼 이름과 값, 정확히 짝을 이룬 괄호 — 그 단순한 규칙이 JSON을 만국 공통어로 만듭니다.
