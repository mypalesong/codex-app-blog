SQLite — 파일 하나가
데이터베이스가 되는 마법

손바닥만 한 ‘수첩’ 한 권과 여러 창구를 가진 ‘은행 창고’에 비유해, 서버 없이 동작하는 데이터베이스의 비밀을 12컷으로 풀어봅니다.

SQLitevsServer DB

데이터베이스라고 하면 서버실 가득한 컴퓨터와 깜빡이는 랙을 떠올리기 쉽습니다. 하지만 어떤 데이터베이스는 그와 정반대의 모습을 하고 있습니다. 이 글에서는 서버 프로그램 하나 없이 파일 하나만으로 완전하게 동작하는 데이터베이스 — SQLite — 가 어떻게 가능한지, 그리고 언제 PostgreSQL이나 MySQL 같은 서버형 데이터베이스로 옮겨가야 하는지를 손바닥만 한 수첩 한 권과 여러 창구를 갖춘 은행 창고에 비유해 하나씩 살펴봅니다.

PART 1 / 3 — 파일 하나의 발견 (컷 01–04)
컷 01

이 작은 수첩 한 권이 데이터베이스라고?

SQLite는 별도의 서버 프로그램 없이, 파일 하나만으로 완전하게 동작하는 데이터베이스다.

데이터베이스라고 하면 서버실 가득한 컴퓨터를 떠올리기 쉽지만, SQLite는 그와 정반대입니다. app.db 같은 파일 하나에 모든 데이터와 구조가 담겨 있어서, 별도의 서버 프로그램을 설치하거나 실행하지 않고도 데이터를 저장하고 조회할 수 있습니다. 손바닥만 한 수첩 한 권이 실제로는 완결된 데이터베이스 역할을 하는 셈입니다.

컷 02

정의부터 명확히: 서버 없이 동작하는 임베디드 데이터베이스

SQLite는 별도의 서버 프로세스 없이 애플리케이션 안에 내장되어 함께 실행되는 임베디드 데이터베이스다.

SQLite는 PostgreSQL이나 MySQL처럼 별도로 켜두어야 하는 서버 프로그램이 없습니다. 대신 앱이 실행되는 동안 SQLite 라이브러리가 그 앱 안에 함께 내장되어 동작하고, 데이터는 지정한 파일 하나에 그대로 저장됩니다. 이 때문에 SQLite는 흔히 embedded database라 불리며, 데이터베이스가 애플리케이션의 일부처럼 함께 실행된다는 뜻을 담고 있습니다.

컷 03

나란히 비교: 수첩과 은행 창고

SQLite는 설치와 설정이 거의 필요 없이 가볍게 시작할 수 있는 반면, PostgreSQL이나 MySQL은 서버 설치와 운영이 필요하지만 다중 접속과 대규모 트래픽에 강하다.

SQLite는 파일 하나만 있으면 되기 때문에 설치 과정이 거의 없어, 프로토타입이나 개인 프로젝트를 빠르게 시작할 때 특히 편리합니다. 반면 PostgreSQL이나 MySQL은 별도의 서버를 설치하고 계속 실행해두어야 하지만, 그만큼 여러 사용자가 동시에 데이터를 읽고 쓰는 상황에 훨씬 강합니다. 작은 개인 수첩과 여러 창구를 갖춘 은행 창고처럼, 규모와 용도에 따라 알맞은 선택이 달라집니다.

컷 04

이 한 문장만 기억해도 왜 수많은 프로토타입이 SQLite로 시작하는지 이해할 수 있다.

WHY IT MATTERS

복잡한 설정 없이 파일 하나만 있으면 데이터를 저장하고 꺼내 쓸 수 있다는 것, 그것이 SQLite의 핵심입니다. 이 한 문장만 기억해도 왜 수많은 모바일 앱과 소규모 웹 프로젝트, 그리고 바이브 코딩 프로토타입이 SQLite로 시작하는지 이해할 수 있습니다.

PART 2 / 3 — 도구와 동작 원리 (컷 05–08)
컷 05

생태계 카탈로그: 수첩을 다루는 도구들

SQLite는 파이썬의 sqlite3 모듈, DB Browser for SQLite, SQLAlchemy 같은 도구들과 함께 쓰이며 데이터를 다루기 쉽게 해준다.

SQLite 파일 자체는 단순하지만, 이를 편하게 다루기 위한 도구는 다양합니다. 파이썬에는 별도 설치 없이 바로 쓸 수 있는 sqlite3 모듈이 내장되어 있고, DB Browser for SQLite 같은 프로그램을 쓰면 코드 없이도 표를 눈으로 보고 편집할 수 있습니다. SQLAlchemy나 Prisma 같은 ORM을 함께 쓰면 SQL 문법을 몰라도 파이썬이나 자바스크립트 코드만으로 데이터를 다룰 수 있습니다. 파일은 하나지만, 그 파일을 다루는 방법은 여러 갈래로 열려 있는 셈입니다.

컷 06

동작 원리: 요청이 파일 한 줄로 저장되기까지

앱에서 데이터를 저장하라는 요청이 오면, SQLite 라이브러리가 그 요청을 해석해 같은 프로세스 안에서 파일에 직접 읽고 쓴다.

앱 코드→ SQL 명령 실행→ 네트워크 없음→ SQLite 라이브러리→ 같은 프로세스→ .db 파일에 기록

앱에서 데이터를 저장하거나 조회하라는 SQL 명령을 내리면, SQLite 라이브러리는 이를 네트워크를 거치지 않고 같은 프로세스 안에서 바로 처리합니다. 서버에 요청을 보내고 응답을 기다리는 과정 없이, 라이브러리가 직접 파일을 열어 필요한 부분을 읽거나 새로운 내용을 적어 넣습니다. 이 짧은 경로 덕분에 SQLite는 별도의 통신 지연 없이 빠르게 동작할 수 있습니다.

컷 07

헷갈리는 경계: 가벼운 만큼 동시 접속엔 약할 수 있다

SQLite는 파일 하나를 여러 프로세스가 동시에 쓰기 작업하려 할 때 제약이 있어, 동시 접속자가 아주 많은 서비스에는 적합하지 않을 수 있다.

SQLite는 읽기 작업은 여러 곳에서 동시에 할 수 있지만, 같은 파일에 쓰기 작업을 할 때는 한 번에 하나의 쓰기만 허용하는 구조를 가지고 있습니다. 사용자가 몇 명뿐인 개인 프로젝트나 프로토타입에서는 전혀 문제가 되지 않지만, 수천 명이 동시에 데이터를 쓰는 서비스에서는 이 제약이 병목이 될 수 있습니다. 그래서 트래픽이 커지면 PostgreSQL이나 MySQL처럼 동시 접속에 최적화된 서버형 데이터베이스로 옮겨가는 경우가 많습니다. database is locked

컷 08

코드로 보는 차이: 몇 줄로 여는 데이터베이스

SQLite는 별도의 설치나 연결 설정 없이, 파일 경로 하나만 지정하면 곧바로 데이터베이스에 연결해 사용할 수 있다.

SQLite — 연결하기
# 파일 경로 하나만 있으면 끝
import sqlite3
conn = sqlite3.connect("app.db")
# 파일이 없으면 자동으로 생성된다
서버형 DB — 연결하기
# 호스트, 포트, 계정까지 필요
import psycopg2
conn = psycopg2.connect(
  host="db.example.com", port=5432,
  user="admin", password="****"
)

SQLite에 연결할 때는 sqlite3.connect("app.db")처럼 파일 경로 하나만 지정하면 끝입니다. 그 파일이 없으면 자동으로 새로 만들어지기까지 합니다. 반면 서버형 데이터베이스는 호스트 주소, 포트 번호, 아이디와 비밀번호까지 갖춰야 연결할 수 있습니다. 바이브 코딩으로 빠르게 아이디어를 시험해볼 때 SQLite가 자주 첫 선택지가 되는 이유가 바로 이 연결 과정의 간단함에 있습니다.

PART 3 / 3 — 실전 감각과 요약 (컷 09–12)
컷 09

흔한 실수: 큰 서비스까지 SQLite 하나로 밀어붙이기

프로토타입 단계를 넘어 사용자가 많아지는데도 계속 SQLite만 고집하면, 동시 쓰기 제약 때문에 서비스 전체가 느려지거나 오류가 날 수 있다.

⚠ writes queue up under load

SQLite로 빠르게 만든 프로토타입이 예상보다 많은 사용자를 끌어모으기 시작하면, 동시에 몰리는 쓰기 요청 때문에 database is locked 같은 오류가 자주 발생할 수 있습니다.

✓ move to a server-based DB in time

이런 신호가 보이기 시작하면 데이터베이스 자체를 바꿔야 할 시점이 다가온 것입니다. 처음부터 큰 규모를 노린다면 서버형 데이터베이스로 시작하는 것도 방법이지만, 작게 시작해 검증한 뒤 필요할 때 옮겨가는 것도 충분히 합리적인 전략입니다.

작은 수첩 한 권 앞에 늘어선 긴 줄을 상상해 보세요. 수첩의 몸집은 그대로인데 줄만 계속 길어진다면, 이제는 여러 창구를 가진 은행 창고로 옮겨갈 시점입니다.

컷 10

트레이드오프 비교표: 수첩과 은행 창고의 규칙집

SQLite는 간편함과 가벼움에서 강점을 갖지만, 동시 접속 처리와 대규모 트래픽 대응에서는 서버형 데이터베이스에 비해 한계가 있다.

great to start, not always to scale
비교 축 SQLite (수첩) 서버형 DB (은행 창고)
설치 / 설정파일 하나로 즉시 시작서버 설치 · 계정 설정 필요
동시 쓰기 처리한 번에 하나의 쓰기만 허용여러 창구에서 동시 처리
배포 형태파일 하나 복사로 배포서버 운영 · 백업 체계 필요
적합한 규모개인 프로젝트 · 프로토타입다중 사용자 서비스
이전 용이성ORM 사용 시 서버형으로 이전 수월—

SQLite는 설치 과정 없이 즉시 시작할 수 있고, 데이터베이스 전체가 파일 하나라서 복사나 백업도 간단합니다. 다만 동시에 많은 쓰기 요청이 몰리는 상황에는 서버형 데이터베이스만큼 강하지 않아서, 사용자가 적은 개인 프로젝트나 프로토타입에 특히 잘 맞습니다. 다행히 SQLite로 시작한 프로젝트도 나중에 SQLAlchemy 같은 ORM을 함께 써두면, 서비스가 커졌을 때 PostgreSQL 같은 서버형 데이터베이스로 비교적 수월하게 옮겨갈 수 있습니다.

컷 11

실전 활용: DB Browser로 파일 속을 눈으로 확인하기

SQLite 파일은 DB Browser for SQLite 같은 무료 도구로 열어 코드 없이도 안의 데이터를 표 형태로 직접 확인할 수 있다.

  • 표 형태로 행/열 확인

  • 값이 예상과 다르게 보일 때

바이브 코딩 중 AI가 만들어준 코드가 데이터를 제대로 저장하고 있는지 궁금하다면, DB Browser for SQLite 같은 무료 프로그램으로 .db 파일을 직접 열어볼 수 있습니다. 코드를 몰라도 표를 보듯 데이터를 확인하고, 필요하면 값을 직접 클릭해 수정할 수도 있습니다. 파일을 눈으로 직접 열어보는 습관은, 지금 만들고 있는 기능이 실제로 의도대로 동작하는지 빠르게 검증하는 좋은 방법입니다.

컷 12

판별법과 핵심 요약: 파일 하나로 끝나면 SQLite다

데이터베이스를 쓰기 위해 별도로 실행해야 하는 서버가 없고 파일 하나로 모든 것이 끝난다면, 그것이 SQLite를 알아보는 가장 쉬운 판별법이다.

SQLite를 알아보는 가장 쉬운 방법은 이것입니다. 데이터베이스를 쓰기 위해 따로 실행해야 하는 서버 프로그램이 없고, .db 파일 하나만 있으면 모든 것이 끝난다면 그것이 SQLite입니다. 앱과 함께 내장되어 실행되기 때문에 가볍고 설정이 간단하지만, 동시에 몰리는 쓰기 작업에는 한계가 있어 규모가 커지면 서버형 데이터베이스로 옮겨가는 것을 고려해야 합니다.

서버 없이 파일 하나로 동작 앱 안에 내장되어 실행 가볍고 설정이 간단함 동시 쓰기엔 제약 있음 커지면 서버형 DB로 이전 고려 판별법: 서버 없이 파일 하나면 SQLite
작은 수첩 한 권, 그리고 은행 창고로 가는 길

SQLite는 서버 없이 파일 하나로 데이터를 담아내는 가볍고 자족적인 데이터베이스입니다.
처음에는 손바닥만 한 수첩 한 권으로 충분하지만, 사용자가 늘고 동시 쓰기가 많아지면 여러 창구를 갖춘 은행 창고, 즉 서버형 데이터베이스로 옮겨가야 할 때가 옵니다.
